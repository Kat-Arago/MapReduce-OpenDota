# MapReduce using the OpenDota Dataset

The rapid growth of data usage for decision-making has created a need for new frameworks to support scalable processing. This has been a factor many analysts consider, especially when handling large amounts of data that require multiple cleaning and processing steps to produce insights that greatly contribute to an entity’s success. In this study, Scalable preprocessing is introduced alongside MapReduce to optimize the efficient processing of big data.
The frameworks were tested using a raw 5GB subset of Dota 2 match data from OpenDota, where epoch timestamps and durations were converted into analysis-ready features. An important part of the methodology is a reduction strategy that leverages partitioned Parquet files as storage, arranging data to enable efficient partition pruning. This method also employed the described MapReduce workflow, focusing on computation locally and reducing I/O overhead.
Analytical utility saw notable enhancements based on empirical results, with the highest match activity observed in 2015. Over time, the average match duration has also increased. Together, these outcomes demonstrate the framework’s capability to handle high-velocity, production-grade pipelines without sacrificing row retention or query performance. In summary, Scalable Processing establishes a robust foundation for integrating large-scale preprocessing and downstream machine learning operations in data-intensive settings.

# Methodology
MapReduce showcases the Spark application pipeline using a groupBy aggregation. According to Holdsworth and Kosinski, MapReduce is a model for processing massive volumes of data in parallel to speed up processing. In this case, it is used to split large datasets into smaller chunks or files so Spark can run partitions in parallel, making the process scalable and preventing bottlenecks.
  ## i. Cleaning
The cleaning methodology is a foundational component of the Scalable Processing framework, designed to handle high-velocity data without sacrificing row retention. During this phase, which is integrated into the Map phase of the Spark pipeline, the system reads raw match records and parses individual fields to handle them independently and locally within each partition. A primary focus of the cleaning and preparation process is feature derivation, where raw epoch timestamps are converted into analysis-ready years and months, and match durations are standardized into minutes. These steps ensure that the data is correctly formatted and optimized for subsequent aggregation, preventing potential errors or "financial damage" that could arise from incorrect key locations or misplaced records during the later stages of the workflow.
  ## ii. Transformation
The methodology for Transformation within the Scalable Processing framework focuses on converting raw data into analysis-ready features to enhance analytical utility. This process primarily occurs during the Map phase of the Spark pipeline, where the system reads individual match rows and parses specific fields. Key transformation activities include feature derivation, such as extracting the year and month from raw epoch timestamps and converting match durations into minutes. By performing these computations locally within each partition, the framework ensures the rows are properly prepared for subsequent grouping and aggregation. These transformations are essential for generating the final data frame, which allows the researchers to track metrics like the average win rate per team and match duration trends over time.
  ## iii. Reduction
The reduction methodology in the report focuses on optimizing storage and computational efficiency for large-scale data. This strategy leverages partitioned Parquet files as the primary storage format, which allows the system to perform partition pruning. By arranging the data this way, the framework can skip irrelevant data segments during queries, significantly reducing I/O overhead and focusing computation locally. This reduction process ensures that the system maintains high-velocity, production-grade pipelines without sacrificing row retention or query performance.
